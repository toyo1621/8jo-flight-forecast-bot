# 非機能要件スコアカード

このスコアカードは、現在の設計を9点水準へ近づけるための基準です。点数は固定値ではなく、運用状況と事故履歴に応じて見直します。

| 観点 | 現在の目安 | 現在の主な対策 | 9点水準の条件 |
| --- | ---: | --- | --- |
| 可用性 | 9/10 | GitHub Pages静的配信、前回予報キャッシュ、6時間ごと更新、workflow timeout | 外部API障害時も公開ページが維持され、失敗が通知され、復旧手順が明文化されている |
| 拡張性 | 9/10 | `app_config.py`、`presentation.py`、BigQuery backend | 新しい便・モデル・表示項目を局所変更で追加できる |
| 保守性 | 9/10 | README/README_AI、テスト、責務分離、Issueテンプレート | 仕様・設定・運用がドキュメント化され、変更時チェックリストがある |
| テスト容易性 | 9/10 | pytest、CI、データ品質テスト、依存更新に強いworkflowテスト | PRごとに自動テストが走り、主要な境界値と失敗時挙動を検証する |
| セキュリティ | 9/10 | GitHub Secrets、Workload Identity、Dependabot、`SECURITY.md` | 鍵を保存せず、依存更新が自動検知され、権限が最小化されている |
| 監視性 | 9/10 | Actions badge、Actions summary、Data Quality Report、成果物保存 | 更新失敗・データ異常・API障害をGitHub上で追跡できる |
| 運用性 | 9/10 | `docs/operations.md`、手動実行手順、品質チェックコマンド、運用障害Issue | 誰でも同じ手順で確認・復旧・再生成できる |
| 性能 | 9/10 | 静的HTML、軽量CSS/JS、pip cache | 通常閲覧はCDN配信で高速、生成処理もActions内で完了する |
| コスト効率 | 9/10 | GitHub Pages中心、BigQuery小規模利用、常時サーバーなし | 常時サーバー不要で、BigQuery/API呼び出しも更新時中心に抑える |
| データ品質 | 8/10 | `data_quality.py`、BigQuery/SQLiteチェック、データ修正Issue | 重複・未知ステータス・欠航理由欠損・気象欠測を継続的に検出し、既知欠損の修正フローがある |
| 法務・コンプライアンス | 9/10 | 気象業法への注意書き、参考値明記、公式判断ではない旨の表示 | 公式予報/航空会社判断と誤認されない表示と運用判断を維持する |
| ユーザビリティ | 9/10 | 色凡例、確率記号、見方ガイド、詳細画面 | 非専門家が色・記号・モデル別参考値・天候信頼度をページ上で理解できる |

## 次に9点へ近づける候補

- BigQuery上の既知欠損（欠航理由・視程・一部日の便欠落）の継続修正
- GitHub Actions失敗時の外部通知（Slack/LINE/メール等）
- BigQueryデータ品質の定期トレンド化
- 主要画面のスクリーンショット回帰テスト
- Cloud Run移行時のCloud Monitoringアラート
- データ修正用の管理画面と認証
